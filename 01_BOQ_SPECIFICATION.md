# مواصفات نظام الحصر والكميات (BOQ) — مكتب رامون الفني للتشطيبات والمقاولات

> هذا الملف هو "المرجع الفني" (Source of Truth) لكل بنود الحصر، متغيراتها، ووحدات قياسها.
> كل بند مكتوب بصيغة تسمح لمهندس البرمجة بتحويله مباشرة إلى نموذج بيانات (Data Model) في Firestore + واجهة إدخال (Form) ديناميكية.

---

## 0. فكرة عامة عن منطق النظام

### 0.1 خيارات بداية المشروع (Project Type Wizard)
عند إنشاء مشروع جديد، يسأل النظام:

1. **نوع العمل**:
   - إنشاء جديد (Core & Shell كامل من الصفر)
   - تشطيب فقط (المبنى/الشقة جاهزة معماريًا، التشطيب من الصفر)
   - تجديد / تشطيب جزئي (Renovation) — فيه بنود قديمة تتشال (هدم/فك) وبنود جديدة تتركب

2. **هل يوجد تعديل معماري؟** (نعم/لا)
   - لو "نعم" → يتم تفعيل قسم "أعمال الهدم والمباني" بالكامل (هدم حوائط، بناء حوائط جديدة، نقل ردم)
   - لو "لا" → يتم تفعيل فقط بنود فك التشطيبات القديمة (لو تجديد) بدون هدم مباني

3. **هل المشروع يحتاج تأسيسات جديدة بالكامل أم تعديل/إضافة على تأسيسات قائمة؟**
   - تأسيس كامل (سباكة + كهرباء + تكييف + تيار خفيف من الصفر)
   - تعديل/إضافة جزئية على تأسيس قائم (يتم تحديد البنود المطلوبة فقط)
   - بدون تأسيس (تشطيب فوق تأسيس قائم سليم)

بناءً على الإجابات الثلاثة، النظام يُفعّل (Show/Hide) الأقسام والبنود المناسبة تلقائيًا من الشجرة الكاملة أدناه، مع إمكانية للمهندس المسؤول أن **يضيف بندًا غير موجود** أو **يحذف بندًا موجود** يدويًا من أي مشروع (Custom Override).

### 0.2 الهيكل العام لكل بند (Item Schema)
كل بند في الشجرة بيتبني بنفس البنية دي (Generic BOQ Item Schema):

```json
{
  "id": "string (auto)",
  "section": "string (اسم القسم الرئيسي)",
  "subsection": "string (اسم البند الفرعي)",
  "title": "string (اسم البند كما يظهر للعميل/المهندس)",
  "applicableTo": ["new_build","finishing_only","renovation"],
  "requiresArchModification": "boolean",
  "unit": "string (م² / م³ / متر طولي / نقطة / عدد / يوم / مقطوعية)",
  "quantitySource": "manual | calculated",
  "calculationFormula": "string (لو calculated، يشير لمصدر القياس - مثال: area.livingRoom.wallArea)",
  "specs": [
    {
      "key": "string",
      "label": "string (يظهر للمستخدم)",
      "type": "select | number | text | color | multi-area",
      "options": ["array لو select"],
      "unit": "string (لو number، مثل لتر/م²)",
      "defaultValue": "any",
      "perAreaOverride": "boolean (هل يسمح بقيمة مختلفة لكل مساحة/غرفة؟)"
    }
  ],
  "pricing": {
    "mode": "materials_labor_split | lump_sum | daily_rate",
    "materialUnitPrice": "number",
    "laborUnitPrice": "number",
    "lumpSumPrice": "number",
    "dailyRate": "number",
    "estimatedDays": "number | formula"
  },
  "notes": "string (ملاحظات حرة)",
  "egyptianCodeRef": "string (مرجع الكود المصري إن وجد)"
}
```

> ملاحظة مهمة: حقل `perAreaOverride: true` هو أساس فكرتك في مثال الدهان — أي بند فيه ده، يقدر المهندس يضيف "مساحة" (غرفة/جدارية) ويحدد لها مواصفات مختلفة (لون/خامة/معدل استهلاك) عن باقي المساحات في نفس البند.

### 0.3 آلية "المساحات" (Areas/Zones)
لكل مشروع، يتم تعريف "مساحات" (Zones) أولًا:
- اسم المساحة (مثال: ريسبشن، غرفة نوم ماستر، مطبخ، حمام 1...)
- مساحة الأرضية (م²)
- مساحة الحوائط (م²) — تُحسب تلقائيًا: (محيط الغرفة × ارتفاع الحائط) − (مساحات الأبواب والشبابيك)
- مساحة السقف (م²) = مساحة الأرضية
- الارتفاع (متر) - افتراضي 3.0م قابل للتعديل

هذه المساحات تُستخدم كمصدر لحساب الكميات (`calculated quantitySource`) في كل البنود التي تعتمد على مساحات (دهانات، أرضيات، تكسيات، أسقف معلقة، محارة...).

## 1. مرحلة التأسيس (Foundation Phase)

### 1.1 أعمال الهدم والمباني (التعديل المعماري)
**يظهر فقط لو `requiresArchModification = true`**

| البند | الوحدة | مصدر الكمية | المواصفات القابلة للتعديل | ملاحظات |
|---|---|---|---|---|
| هدم حوائط طوب طفلي/أحمر | م² | manual | سمك الحائط (12 / 25 سم)، نوع الطوب (أحمر/خفان/طوب مصمت) | يُحسب الناتج كردم (م³) تلقائيًا = المساحة × السمك |
| هدم أرضيات/تكسيات قديمة | م² | calculated (من مساحة الأرضيات المطلوب هدمها) | نوع التكسية القديمة (سيراميك/بورسلين/رخام/باركيه) | يشمل سعر النقاشة |
| فك أسقف معلقة قديمة | م² | manual/calculated | نوع السقف القديم | |
| فك تكسيات حوائط قديمة | م² | manual/calculated | نوع التكسية | |
| بناء حوائط جديدة | م² | manual | نوع الطوب (أحمر فرشة/أحمر قائم/خفان/بلوك)، سمك الحائط (12/20/25 سم)، نوع المونة (سن أسمنت 1:5 / 1:4) | حسب الكود المصري لأكواد البناء — نسبة المونة قابلة للتعديل |
| أعمدة/كمرات إضافية (لو وجدت) | متر طولي / عدد | manual | مقاس الحديد (8مم/10مم/12مم)، نسبة الخرسانة (B250/B300) | يحتاج موافقة مكتب استشاري — حقل تنبيه (Warning Flag) |
| نقل المخلفات والردم | م³ | calculated = إجمالي (هدم الحوائط + هدم الأرضيات + هدم الأسقف بنسبة تحويل) | عدد العربات/الديكوتاج، سعر العربة (مقطوعية أو بالم³) | يُحسب تلقائيًا من إجمالي بنود الهدم أعلاه × معامل تحويل (افتراضي 1.0 قابل للتعديل) |
| فك وتركيب أبواب/شبابيك مؤقت (لو تم نقلها) | عدد | manual | نوع الباب/الشباك | |
| سد فتحات قديمة (أبواب/شبابيك) | عدد / م² | manual | نوع الطوب المستخدم في السد | |

**التسعير**: كل بند يدعم `materials_labor_split` (سعر الطوب/الأسمنت + سعر العمالة) أو `daily_rate` (عدد أيام × سعر يومية "حداد/مباني/عتال").

### 2.10 الوجه النهائي للدهانات

> هذا هو البند المثالي لتوضيح فكرة `perAreaOverride` (مثال الدهان بتاعك بالضبط):

| الحقل | النوع | الوصف |
|---|---|---|
| نوع الدهان | select لكل مساحة | بلاستيك مطفي / بلاستيك نصف لمعة / سمكو / دهان زيتي / دهان خارجي |
| كود اللون | text/color picker لكل مساحة | مثال: RAL 9003 أو كود مورد الدهان |
| معدل الفرد (م²/لتر) | number لكل نوع دهان | يحدد افتراضيًا حسب نوع الدهان (مثال: بلاستيك = 10-12 م²/لتر) ويمكن تعديله يدويًا |
| عدد الأوجه | number لكل مساحة | افتراضي 2 |
| **تعدد الألوان لكل مساحة** | array of (sub-area + color + finish) | يسمح للمهندس بتقسيم نفس الغرفة لأكثر من حائط/مساحة بلون/خامة مختلفة (مثال: حائط مميز Accent Wall بلون مختلف) |

**معادلة الحساب الأوتوماتيك**:
```
كمية الدهان (لتر) = (مساحة الحائط أو السقف بالم² × عدد الأوجه) ÷ معدل الفرد (م²/لتر)
عدد العلب = كمية الدهان (لتر) ÷ سعة العلبة (افتراضي 18 لتر، قابل للتعديل)
```

**ملاحظة**: نفس هذا الـ Schema (تعدد القيم لكل مساحة + معدل استهلاك + كود) يُطبَّق بنفس المنطق على:
- بند 2.4 (دهان الحوائط الخارجية لو موجود)
- بند 1.5 (لو فيه أكثر من نوع محارة)
- بند 2.16 (تعدد أنواع الأرضيات لكل غرفة)
- بند 2.8 (تعدد درجات حرارة الإضاءة لكل غرفة)

### 2.11 تركيب الأرضيات (سيراميك / بورسلين / HDF / SPC)

| الحقل | النوع | الوصف |
|---|---|---|
| نوع الأرضية لكل مساحة | select (per area) | سيراميك / بورسلين / رخام / باركيه HDF / SPC / فينيل |
| المقاس | select | 60×60 / 80×80 / 30×60 / 120×60 ... |
| الكود/الموديل واللون | text | |
| نوع الفرز (Rectified/غير Rectified) | select | يؤثر على عرض الفاصل (Grout) |
| عرض الفاصل (مم) | number | |
| نوع السكيرتنج (Skirting) | select | نفس خامة الأرضية / رخام / أسود مطفي / مخفي (Shadow Gap) — مرتبط بمشروع الإضاءة عندك |
| ارتفاع السكيرتنج (سم) | number | |
| نسبة الهالك (Wastage %) | number | افتراضي 10% يُضاف على الكمية المحسوبة |

**معادلة الحساب**:
```
كمية البلاط (م²) = مساحة الأرضية (per area) × (1 + نسبة الهالك%)
عدد البلاطات = كمية البلاط ÷ مساحة البلاطة الواحدة (حسب المقاس المختار)
متر السكيرتنج = محيط الغرفة − عرض الأبواب
```

## 3. منطق "التجديد" (Renovation Logic)

عند اختيار "تجديد" في بداية المشروع، يضيف النظام لكل بند علم إضافي:

```json
{
  "renovationAction": "keep | remove_only | remove_and_replace | new_addition"
}
```

- **keep**: البند موجود ولن يُلمس (لا يُحسب في الـ BOQ)
- **remove_only**: فك/هدم فقط (يُحسب في قسم الهدم — بند 1.1)
- **remove_and_replace**: فك القديم + تركيب جديد (يظهر في قسمين: الهدم + التشطيب الجديد)
- **new_addition**: بند جديد كليًا (لم يكن موجودًا من قبل)

هذا يسمح بإنشاء "حصر تجديد" يوضح بالتفصيل: إيه اللي هيتشال، إيه اللي هيتضاف، وإيه اللي هيفضل كما هو — مع تسعير كل حالة بشكل مستقل.

## 5. الأدوار والصلاحيات (Roles & Permissions)

| الدور | الصلاحيات |
|---|---|
| **Admin (صاحب المكتب)** | كل الصلاحيات + إدارة المستخدمين + إدارة قاعدة بيانات الأسعار العامة (Price List) |
| **مهندس حصر/كميات (Quantity Engineer)** | إنشاء/تعديل المشاريع، تعديل الكميات والمواصفات، رؤية التسعير |
| **مهندس مكتب فني (Technical Office)** | تعديل المواصفات الفنية فقط، رؤية الكميات، تعديل محدود على التسعير (حسب صلاحية) |
| **مهندس مشارك (Collaborator)** | يُضاف لمشروع معين فقط، يقدر يعدل بنود محددة (Section-level permission) |
| **عميل (Client - عبر رابط فقط)** | عرض فقط (Read-only) — لا يرى الأسعار التفصيلية لو مش مطلوب، أو يرى التقرير النهائي فقط بدون تعديل |

### رابط العميل (Client Share Link)
- لينك فريد لكل مشروع (مثال: `ramondoffice.com/view/RMD-2026-001?token=xxxx`)
- يعرض نسخة "للعرض فقط" (Read-only Snapshot) — تحديث تلقائي أو "تجميد" نسخة معينة (Snapshot) حسب اختيار المهندس
- يمكن تحديد: هل يظهر للعميل الأسعار التفصيلية لكل بند، أم فقط الإجمالي العام، أم بدون أسعار نهائيًا (للعرض الفني فقط)

## 6. مخرجات PDF المطلوبة

### 6.1 PDF تقرير العميل
- بيانات المشروع (Header)
- شجرة البنود مجمعة حسب الأقسام (تأسيس / تشطيب)
- لكل بند: الوصف + الكمية + الوحدة (الأسعار حسب صلاحية العرض المحددة)
- الإجمالي العام لكل قسم + الإجمالي الكلي
- تذييل الصفحة: شعار "رامون للتشطيبات والمقاولات" + بيانات التواصل

### 6.2 PDF مواد الشراء (لكل قسم)
- قائمة بكل "المواد" المطلوبة لكل بند (مستخرجة من حقول `specs` التي نوعها مادة)
- الكمية المطلوبة من كل مادة (مع نسبة هالك)
- اسم المورد المفضل (حقل اختياري لكل مادة)
- حالة الشراء (لم يُطلب / تم الطلب / تم التوريد) — Checklist تفاعلي

### 6.3 PDF الـ BOQ الكامل (للاستخدام الداخلي)
- كل البنود + الكميات + أسعار المواد + أسعار العمالة + عدد أيام التنفيذ + الإجمالي
- إجمالي مواد فقط / إجمالي عمالة فقط / الإجمالي الكلي
- جدول زمني تقديري (Gantt مبسط) بناءً على `estimatedDays` لكل بند بشكل تراكمي
